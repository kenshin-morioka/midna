# 0002 - オフライン動作を基本とし、ブラウジング系ツールのみ外部通信を許可

- Status: Accepted
- Date: 2026-05-18
- Deciders: kenshin-morioka

## Context

design doc v0.1 の "Core Philosophy / Local-first" は「ローカル優先・クラウドはオプション」と書かれているが、ユーザーは「wifi 不要で動く AI エージェントが欲しい」と明示している。"Local-first" の解釈を厳密化し、どの動作が外部通信を行ってよいかを明文化する必要がある。

評価軸:

- プライバシー（自分専用ツールなので会話履歴をクラウドに送らないこと）
- 通信なしで完結する保証（オフライン環境でも基本動作が成立すること）
- 最新情報を扱えること（ブラウジング用途は本質的にネット必須）
- 初期セットアップの実用性（モデル pull など最初の 1 回はネットが要る）

## Decision

- **推論はオフラインで完結する。** LLM 推論呼び出し（chat / generate）は完全にローカルランタイム（Ollama 等）で行い、外部 API は呼ばない。
- **例外: ブラウジング系ツールは外部通信を許可する。** web fetch / web search 等、最新情報取得を本質とするツールに限ってネット通信を許可する。
- **初回セットアップに必要な通信は許容する。** ランタイムのインストール、モデルファイルの pull、Cargo crate の取得など、セットアップ作業は通信が前提でよい。実行時の通信ポリシーとは区別する。

## Rationale

- 安全性とプライバシーを重視する個人ツールなので、会話本体はローカル完結が望ましい。
- 一方で「最新情報を聞きたい」という本質的にネットが要る用途を完全に閉ざすと実用性を損なうため、明示的に穴を空ける範囲を限定する。
- 初回セットアップは「実行時の通信」とは別レイヤとして扱うのが自然（ビルド時通信を禁止すると現実的に作業できない）。

## Consequences

- Positive
  - 通信なしで会話できる明確な保証 → 飛行機・移動中など接続不安定環境でも使える
  - 推論ログがクラウドに残らない → プライバシー
  - クラウド API コスト 0
- Negative / Trade-offs
  - 初回モデル pull とランタイムインストールは通信が必須（オフライン環境では事前準備が要る）
  - クラウド最高峰モデル（フロンティアモデル）の品質に追従するには、別途明示的に切り替える仕組みが必要になる
  - ブラウジング系ツールの実装と「許可された通信先」のポリシー定義が必要になる（permission system 側で扱う）

## Alternatives Considered

- **クラウド API 併用前提** — Anthropic / OpenAI を一級扱いにする案。品質と速度では有利だが、ユーザーの「ローカルで動かしたい」という明示要件と相反するため却下。
- **完全オフライン（ブラウジングも禁止）** — 最も厳格だが、最新情報取得という重要ユースケースを失う。実用性が著しく落ちるため却下。
- **個別ツールごとに通信ポリシーを後で考える** — 方針が定まらないまま実装が進むと一貫性が失われるため、v0 段階で明文化することを選択。
