# 0001 - 実装言語に Rust を採用

- 状態: 採択
- 日付: 2026-05-18
- 決定者: kenshin-morioka

## 結論

midna の実装言語に **Rust** を採用する。

## 背景

midna は「自分専用の、手元で動く AI エージェント」を目指す CLI ツール。Ollama などの手元の LLM ランタイムと HTTP でやり取りし、将来はファイル操作・シェル実行・Web からの情報取得といったツールを介して動かす予定。最初に実装言語を決める必要がある。

判断の軸:

- 動作の速さ（LLM 呼び出しの取りまとめや、将来複数エージェントを並行で動かすときの性能）
- 安全に動かせること（権限、メモリの扱い、データの扱い）
- 特定の LLM ランタイム（Ollama など）に縛られず、別のものへ乗り換えやすいこと
- LLM や AI まわりの周辺ライブラリの厚さ
- 配布のしやすさ（自分用のツールなので、1 個のバイナリで配れると嬉しい）

## 理由

- **動作の速さを優先したい。** 可読性は良いが Python は速さの面で不利が大きい。
- **特定のランタイムに縛られたくない。** Go は Ollama 本体と相性が良いが、将来 Ollama より良い手元向けランタイムが出てきたとき、Go を選んだ優位性は薄れる。手元ランタイムに合わせて言語を寄せたくない。
- **手元で動かす AI エージェントとして、安全に動かせることが個人的に大事。** Rust の所有権の仕組みとメモリの安全性は、将来ツールがファイル操作やシェル実行に手を伸ばすときの不安を減らせる。
- **複数のエージェントを並行して動かす将来像があり、Rust なら安全に並行処理が書ける。** `Send` / `Sync` の制約と async/await（tokio）が土台として向いている。

## 影響

- メリット
  - 動作が速く、メモリの使用量も少ない
  - 1 個のバイナリで配布できる
  - メモリ安全 + `Send` / `Sync` による並行処理の安全保証
  - async/await を使って複数エージェントを並行に動かす素地ができる
- デメリット
  - AI / LLM まわりの周辺ライブラリは Python ほど揃っていない（薄い HTTP 呼び出しを自前で書く場面が増える）
  - ビルドにかかる時間や開発の立ち上げの速さは、スクリプト言語より遅い
  - 借用チェッカの学習コスト（自分用の開発なので許容範囲）

## 他に検討した案

- **Python** — AI / LLM まわりの周辺ライブラリが一番厚く、開発の立ち上げは速い。しかし動作の速さで不利、並行処理の制約（GIL）もあるため不採用。
- **Go** — 1 バイナリで配れて並行処理が書きやすい。Ollama 本体との相性も良い。ただし Ollama 依存度が下がる将来でその優位性が薄れるため不採用。
- **TypeScript (Node)** — Claude Code などと同じ土台で、MCP との連携も組みやすい。だが手元で動く CLI を主軸に置きたいので、Node ランタイムを同梱して配るコストを避けたく不採用。
