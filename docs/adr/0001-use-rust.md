# 0001 - 実装言語に Rust を採用

- Status: Accepted
- Date: 2026-05-18
- Deciders: kenshin-morioka

## Context

midna は「自分専用のローカル AI エージェント」を目指す CLI ツール。Ollama などのローカル LLM ランタイムと HTTP で対話し、tools を介して filesystem / shell / web fetch 等を扱うことを将来含めて計画している。最初に実装言語を確定させる必要がある。

評価軸:

- 処理速度（推論呼び出しのオーケストレーション、並列エージェント稼働を見据えた性能）
- ローカル AI エージェントとしての安全性管理（権限、メモリ安全、データ取り扱い）
- 特定 LLM ランタイム（Ollama 等）への依存を将来切り替え可能にしたい
- LLM/AI 周辺エコシステムの厚さ
- 配布のしやすさ（個人ツールなのでシングルバイナリ性は嬉しい）

## Decision

**Rust** を採用する。

## Rationale

- **速度を重視したい。** 可読性は良くとも Python は速度面の不利が大きいため避ける。
- **provider ロックインを避けたい。** Go は Ollama 本体実装と相性が良い候補だが、将来 Ollama 以外のより最適なオープンモデル/ランタイムが出てきたとき、Go の優位性は薄れる。特定ランタイムに言語を寄せたくない。
- **ローカル AI エージェントは安全性管理が個人的に重要。** Rust の所有権モデルとメモリ安全性は、tools が filesystem / shell を触る将来を見据えて有利。
- **安全な並列エージェント稼働が期待できる。** 複数エージェントを同時に走らせる将来像があり、Rust の `Send`/`Sync` 制約と async/await（tokio）はその基盤として適している。

## Consequences

- Positive
  - 高速な実行、低メモリフットプリント
  - シングルバイナリ配布
  - メモリ安全 + `Send`/`Sync` による並列性の安全保証
  - async/await による並列エージェント実装の足場
- Negative / Trade-offs
  - AI/LLM 周辺ライブラリのエコシステムは Python ほど厚くない（自前で薄い HTTP クライアントを書く場面が増える）
  - ビルド時間と開発初速がスクリプト言語より遅い
  - 借用チェッカの学習コスト（個人開発なので許容範囲）

## Alternatives Considered

- **Python** — LLM エコシステム最厚で開発初速が高いが、処理速度の不利と並列処理の制約（GIL）が決定打となり却下。
- **Go** — シングルバイナリ性とシンプルな並行性モデル、Ollama 本体との相性は良い。だが Ollama 依存度が下がる将来でその優位性が薄れる懸念があり却下。
- **TypeScript (Node)** — Claude Code 等と同じスタックで MCP 連携の親和性は高いが、本ツールではローカル CLI を主軸に置きたく、Node ランタイム同梱配布のコストを避けたかったため却下。
- **Ruby** — design doc の当初構造 (`exe/midna`, `lib/midna/`) は Ruby gem 風だったが、これは執筆時のレイアウト便宜であり、ローカル LLM 周辺エコシステムの薄さから却下。
