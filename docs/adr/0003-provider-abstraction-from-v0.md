# 0003 - LLM の提供元を差し替えやすい構造を v0 から入れる

- 状態: 採択
- 日付: 2026-05-18
- 決定者: kenshin-morioka

## 結論

LLM の提供元（プロバイダ）を差し替えやすくする共通の窓口を、v0 の時点から入れる。
具体的には Rust の trait `Provider` を切り、Ollama の実装はこの trait 経由で呼ぶ。

```rust
#[async_trait::async_trait]
pub trait Provider: Send + Sync {
    async fn chat(&self, messages: &[Message]) -> Result<Message, MidnaError>;
    fn name(&self) -> &'static str;
}
```

会話のやり取りは `Message { role: Role, content: String }` の形に揃える。

## 背景

v0 で実際に使う LLM の提供元は Ollama 1 個だけ。「実装が 1 個しかないのに最初から共通の窓口を作るのは先回りしすぎではないか」という見方と、「将来別の提供元へ乗り換えたい意向が既にあるのだから、最初から窓口を切るほうがよい」という見方の、どちらを取るかが論点。

将来 Ollama より良い手元向けランタイムが出てきたら乗り換えたい、という方針は最初から持っている。特定の提供元に縛られたくないという考えは、言語選定のとき（[ADR 0001](0001-use-rust.md)）の判断軸にもなった。

## 理由

- 将来の乗り換えを最初から見越している。あとから共通窓口を入れる手戻りより、最初から trait を切るほうが痛みが少ない。
- テストが書きやすくなる（テスト用のダミー実装を差し込める）。
- 窓口を作るコストが小さい。Rust の trait と `async_trait` を使えば、書く量はごくわずか。
- 複数のエージェントを並行で動かす将来像があるので、提供元が `Send + Sync` であることを型で保証しておく価値がある。

## 影響

- 良い点
  - 別の提供元（llama.cpp を直接呼ぶ、Anthropic、OpenAI、別の手元向けランタイム）を追加するときに、新しいファイルを 1 個足すだけで済む
  - `Send + Sync` の制約により、並行に動かす際の安全性が型で守られる
  - テストでダミー実装に差し替えられる
- 引き換え / 困る点
  - 実装が 1 個しかない段階での共通窓口は「先回りで作り込みすぎ」と見える可能性がある
  - `async_trait` を v0 から依存に入れることになる（trait の async が言語標準で安定したら剥がす予定）

## 他に検討した案

- **Ollama を直接呼ぶ書き方** — `OllamaClient` をセッションが直接呼ぶ最小の構成。コードは短くなるが、提供元を切り替えるときにセッション・CLI・テストを広く書き直す必要が出るため不採用。
- **複数の提供元を v0 から実装する（Ollama と Anthropic など）** — 共通窓口を作る根拠は揃うが、v0 のスコープ（最小の会話ループ）を逸脱するため不採用。共通窓口は入れるが、実装は 1 個に絞る中庸を選んだ。
