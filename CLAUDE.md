# Midna プロジェクトでの作業ルール

**抽象度を上げ、原則だけを書く**。個別ケースを列挙すると局所最適なルールが溜まる。

**同じ事実は 1 箇所だけに書く**。他所からは参照する。ドキュメントにも DRY を効かせる。

## ファイルとディレクトリの責務

ルール / テンプレート / ドキュメントを混ぜない。

| 種別 | 場所 | 責務 | 遵守すること |
| --- | --- | --- | --- |
| ルール | `CLAUDE.md` | このリポジトリでの作業の原則 | 事実・具体・雛形を書かない。原則だけを抽象的に書く |
| テンプレート | `.claude/templates/` | 各ドキュメントの雛形（形式） | 形式の骨格だけ。判断や事実は書かない |
| 設計ドキュメント | `docs/design/<name>.md` | 機能や改修ごとの設計 | 1 機能 1 ファイル。判断の理由は ADR を参照 |
| 設計判断 | `docs/adr/NNNN-kebab.md` | 方針や引き換えを伴う判断 | 1 判断 1 ファイル。プロダクト全体の方針は ADR の集合で表現する |
| 用語集 | `docs/ubiquitous-language.md` | 用語の正式表記 | 各文書からはここを参照する。他所で再掲しない |
| 入口 | `README.md` | 読み手へのオリエンテーション | 各リンクには「何のリンクか」の一言を添える |

## 指摘は再発防止までを 1 セットで

是正を受けたら、原則を該当ファイルに追記し、リポジトリ内の同型違反を grep して同じターンで直す。

## ドキュメントの形式

- 人向け文書は、日常会話で口にする言葉で書く。普段出てこない専門用語っぽい言葉は避ける。一方で過剰に置き換えない（普段使う横文字や技術固有名はそのまま）
- 図解は基本的に mermaid で書く

## 推測で断定しない

確定的な手がかりがなければ確認する。やむを得ず推測で進めるなら「推測です」と明示する。

## PR は変更の種類で分ける

レビュアーが追うコンテキストを 1 種類に保つため、PR は変更の種類で分ける。ドキュメント変更と実装変更を同じ PR に混ぜない。

## 終わるとき

コミット前に動作確認（`cargo build` / `cargo test`、必要なら `cargo clippy`）→ push → 必要なら PR 作成。
