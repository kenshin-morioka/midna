# midna プロジェクトでの作業ルール

`~/.claude/CLAUDE.md` に上乗せする、このリポジトリ固有のお作法。

**抽象度を上げ、原則だけを書く**。個別ケースを列挙すると局所最適なルールが溜まる。

**同じ事実は 1 箇所だけに書く**。他所からは参照する。ドキュメントにも DRY を効かせる。

## 指摘は再発防止までを 1 セットで

是正を受けたら、原則を該当ファイルに追記し、リポジトリ内の同型違反を grep して同じターンで直す。

## 設計判断は ADR、概観は design doc

方針や引き換えを伴う判断は `docs/adr/NNNN-kebab.md` に残し、design doc にはリンクだけ置く。実装の細かい選択は ADR にしない。

## ADR は結論先頭、自分の主語で、両面を書く

- 上から読んで真っ先に判断が分かる構成にする（結論 → 背景 → 理由 → 影響 → 他に検討した案。雛形は `docs/adr/README.md`）
- 自分の判断は自分の主語で書く。第三者から仕様を受けたかのような書き方をしない
- 「他に検討した案」には事実だけを残す。整合を取り繕って後付けで案を並べない
- 判断の正負両面（メリットとデメリット）を書く

## ドキュメントの形式

- 人向け文書は、日常会話で口にする言葉で書く。普段出てこない専門用語っぽい言葉は避ける。一方で過剰に置き換えない（普段使う横文字や技術固有名はそのまま）
- 図解は mermaid で書く

## 推測で断定しない

確定的な手がかりがなければ確認する。やむを得ず推測で進めるなら「推測です」と明示する。

## 終わるとき

コミット前に動作確認（`cargo build` / `cargo test`、必要なら `cargo clippy`）→ push → 必要なら PR 作成。
