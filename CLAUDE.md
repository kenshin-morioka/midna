# midna プロジェクトでの作業ルール

このファイルは midna リポジトリで Claude Code が作業するときの方針をまとめたもの。
グローバルな `~/.claude/CLAUDE.md` の指示に上乗せする形で、このリポジトリだけのお作法を記録する。

## 運用ルール: 指摘されたら CLAUDE.md に追記 → 既存ファイルを点検

作業中にユーザーから指摘されたこと・是正されたことは、必ず以下を **同じターンで** 実施する。同じ指摘を二度受けないため。

1. このファイル（リポジトリ直下の `CLAUDE.md`）に該当ルールを追記する。
2. **追記したルールに照らして、リポジトリ内の既存ファイルを走査し、同じ違反が残っていないかを点検する**（grep などで網羅的に）。
3. 違反が見つかったら、その場で修正してコミットに含める。
4. 必要に応じてグローバル memory (`~/.claude/projects/.../memory/`) にも保存し、将来のセッションでも引き継げるようにする。

「ルールを書く」だけで終わらせず、**ルール追加と同時に過去の違反箇所も直しに行く**のがこのプロジェクトの作法。

## 設計判断は ADR にまとめる

- 言語選定、ライブラリ選定、運用ポリシーなど **方針や引き換えを伴う判断は `docs/adr/` に ADR として記録する**。
- design doc (`docs/design/*.md`) には判断の中身を書かない。**ADR への相対リンクだけ置く**。
- 命名は `NNNN-kebab-case-title.md`。連番、欠番なし。
- 既存 ADR を覆すときは新しい ADR を作り、古いほうの状態を `Superseded by NNNN` に書き換える。
- 「変数の付け方」「関数の切り出し方」など実装の細かい選択は ADR にしない。

## ADR の書き方

- セクション構成は **結論 → 背景 → 理由 → 影響 → 他に検討した案**（雛形は `docs/adr/README.md`）。「結論」を先頭に置き、上から読んでまず何が決まったかが分かるようにする。
- **本人視点で書く。** 決定者は自分（kenshin-morioka）なので、「ユーザーは○○と明示している」「ユーザーが○○と話している」のような **第三者から仕様を受けている書き方をしない**。主語を省略するか「自分としては○○にしたい」「○○という方針は最初から持っている」と書く。
- **「他に検討した案」には、本気で比較した案だけを書く。** 当初の勘違いや推測ミスで一瞬候補に挙がっただけのものを後付けで並べない（例: midna 0001 の初稿に Ruby を入れたが、design doc のディレクトリ構成からの推測ミスを引きずっていただけで、実際の比較対象ではなかった）。
- 「良い点」と「引き換え / 困る点」を両方書く。良いことだけ書かない。

## 人向け文書は日本語で書く

- ADR、design doc、README、CLAUDE.md、issue / PR の本文などは **日本語で書く**。
- コード中のコメントも、ねらいや背景を説明するものは日本語。

## 普段使わないカタカナ語を避ける

無理に英語そのままを取り込んだ「専門用語っぽいカタカナ語」を文書に混ぜると読みづらくなる。
できる限り **自然な日本語の言い回し** に置き換える。

避ける例 → 言い換え例:

- 「provider ロックイン回避」 → 「特定の提供元（プロバイダ）に縛られないようにする」「乗り換えやすさを残す」
- 「トレードオフ」 → 「引き換え」「代わりに失うもの」「妥協点」
- 「エコシステム」 → 「周辺ライブラリ」「関連する道具立て」
- 「フロンティアモデル」 → 「最新の高性能モデル」
- 「ペイン」 → 「困りごと」「困っている点」
- 「YAGNI」 → 「使う前に作らない」考え方
- 「オーバーエンジニアリング」 → 「先回りで作り込みすぎ」

ただし以下は **そのまま使ってよい**:

- 言語/ランタイム/ライブラリの正式名: Rust, Cargo, tokio, reqwest, Ollama, など
- 言語機能の用語: trait, async/await, Send, Sync, 借用チェッカ, など
- 通信や規格の正式名: HTTP, JSON, MCP, ADR, など

判断に迷ったら、**「日本語の文章として、普段の会話で出てくるか」** を基準にする。
出てこないカタカナ語は言い換えるか、初出で意味を補う。

## 技術スタックなど重要な判断は推測しない

- ディレクトリ名（`exe/` `lib/` など）のような弱いヒントだけで言語やフレームワークを断定しない。
- `Cargo.toml` `package.json` などの確定的な手がかりがなければ、ユーザーに確認する。
- 推測で進めざるを得ない場合は「推測です」と明示する。

## 作業を終えるときの手順

`~/.claude/CLAUDE.md` のルールに従う:

- 作業内容をコミット
- リモートブランチに push
- まだ PR がなければ PR を作成
- コミット前に必ず動作確認（`cargo build` / `cargo test`、必要なら `cargo clippy`）
